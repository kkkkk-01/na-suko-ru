# ESP32 名札型端末 ファームウェア仕様書

Webシミュレーター（client-user）と**完全に同一のAPI仕様**で動作するよう設計する。
本仕様に従って実装すれば、サーバー側の変更なしで実機に置き換え可能。

## 1. ハードウェア構成（想定）

| 部品 | 用途 |
|---|---|
| ESP32-S3 | メインSoC（Wi-Fi + BLE同時利用可） |
| タクトスイッチ | 呼出ボタン（長押し=緊急は将来拡張） |
| I2S MEMSマイク + I2Sアンプ/スピーカー | 双方向通話（Phase2以降） |
| LED (RGB) | 状態表示 |
| リチウムポリマー電池 + 充電IC | バッテリー駆動 |

## 2. 状態LED仕様

| 状態 | LED |
|---|---|
| 正常（オンライン） | 緑 点滅（5秒に1回） |
| 呼出中 | オレンジ 点滅（1秒間隔） |
| 通話中 | 緑 点灯 |
| Wi-Fi切断 | 赤 点滅 |
| バッテリー低下(20%以下) | 赤/オレンジ 交互 |

## 3. BLEビーコンスキャン（位置検知）

```
- スキャン間隔: 15秒ごとに3秒間パッシブスキャン
- 対象: iBeacon形式、施設固有のUUIDでフィルタ
- ビーコンID: iBeaconのMajor/Minorを "BCN-xxx" にマッピング（サーバーのbeaconsテーブルと一致させる）
- 最寄り判定: RSSI移動平均（直近3スキャン）が最大のビーコンを現在地とする
- ヒステリシス: 現在地変更は新ビーコンが +6dB 以上強い場合のみ（ふらつき防止）
- 受信限界: -90dBm 未満はスキャン結果から除外
```

### 3-1. アルゴリズム机上検証結果（シミュレーション済み）

本判定ロジックは client-user の「リアル電波モード」に**同一実装**があり、
パスロスモデル（TxPower=-40dBm, n=2.8, 壁減衰8dB/枚, フェージング±6dB）で検証済み:

| テスト | 結果 |
|---|---|
| 部屋中心での判定正答率（全8ヶ所×100試行） | **100%** |
| 部屋境界での判定ふらつき（ヒステリシスあり） | 0.2回/20スキャン |
| 同（ヒステリシスなし） | 5.0回/20スキャン |
| **ヒステリシス効果** | **ふらつき96%削減** |

→ 移動平均3サンプル + ヒステリシス+6dBの組み合わせを実機でもそのまま採用すること。
実機では n（減衰係数）と壁減衰が環境依存のため、施設での実測でTxPower基準値のみ再キャリブレーションする。

## 4. サーバーAPI仕様（Webシミュレーターと同一）

### 4-1. ハートビート（30秒ごと）
```
POST /api/v1/locations/heartbeat
Headers: Content-Type: application/json
         X-API-Key: <APIキー>
Body: {
  "user_id": 1,              // 端末に紐づく利用者ID（プロビジョニング時に書込）
  "beacon_id": "BCN-101",    // 最寄りビーコン（不明時はnull）
  "rssi": -58,               // 最寄りビーコンのRSSI
  "battery_level": 85        // 電池残量%
}
→ 200 { "ok": true, "server_time": "..." }
```
- **90秒間ハートビートが途絶えるとサーバーがオフライン判定しアラート発報**する
- 送信失敗時は指数バックオフ（1s→2s→4s、最大30s）でリトライ

### 4-2. 呼出（ボタン押下時）
```
POST /api/v1/calls/request
Body: {
  "caller_id": 1,
  "beacon_id": "BCN-TOILET",  // 押下時点の最寄りビーコン
  "rssi": -52
}
→ 201 { "call_id": 12, "status": "ringing", "location": {...} }
```
- ボタン押下は即座にLEDをオレンジ点滅にし、API成功でビープ1回
- **API失敗時は3回リトライ後、赤LED点滅で利用者に「届いていない」ことを明示**（不達の可視化）

### 4-3. 通話（Phase2以降）
- 応答は `call:answered` を MQTT または WebSocket で受信
- 音声: Asterisk (PJSIP/UDP) にSIP登録し、`sip_username`/`sip_password`（devicesテーブル）で認証
- コーデック: G.711 (ulaw) → 帯域・CPUが許せばOpus

## 5. 電源管理

```
- 通常時: Wi-Fi modem sleep + BLEスキャンのデューティ制御で平均消費 ~30mA目標
- ボタン割込: GPIO割込で即復帰（deep sleepは使わない: 呼出遅延<1秒を優先）
- 電池残量: ADCで電圧測定→%変換、ハートビートに載せる
- 20%以下: サーバーが自動でbattery_lowアラート（端末側実装不要）
```

## 6. プロビジョニング（端末初期設定）

1. 初回起動時: SoftAPモード（SSID: `NURSECALL-SETUP-xxxx`）
2. 管理者がスマホで接続し、設定ページで入力:
   - 施設Wi-Fi SSID/パスワード
   - サーバーURL / APIキー
   - user_id（利用者との紐付け）
3. NVSに保存し再起動 → 通常運用開始

## 7. OTA更新（Phase3）

- サーバーの `/api/v1/firmware/latest` （将来実装）をハートビート時に確認
- 段階配信: サーバー側でロールアウト率を制御（1台→10台→全台）
- 更新失敗時は前バージョンに自動ロールバック（ESP32のOTAパーティション機構）
